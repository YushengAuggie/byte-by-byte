## 核心概念 / Core Concepts

### 1. Types vs Interfaces

```typescript
// Type alias — can represent any shape, including unions/intersections
type UserStatus = "active" | "inactive" | "banned";

type User = {
  id: number;
  name: string;
  status: UserStatus;
  email?: string;  // optional field
};

// Interface — for objects and classes; supports declaration merging
interface AdminUser {
  id: number;
  name: string;
  permissions: string[];
}

// Extending
interface SuperAdmin extends AdminUser {
  canDeleteUsers: boolean;
}

// Key difference: types can't be merged, interfaces can
interface Window {
  myCustomProp: string;  // ✅ merges with existing Window type
}
// type Window = { myCustomProp: string }  // ❌ duplicate identifier error
```

**经验规则 / Rule of Thumb:**
- 用 `interface` 定义对象形状（尤其是公共 API）
- 用 `type` 定义联合类型、工具类型、或非对象类型

---

### 2. Generics — 泛型让代码复用

```typescript
// Without generics — duplicated logic
function getFirstNumber(arr: number[]): number {
  return arr[0];
}
function getFirstString(arr: string[]): string {
  return arr[0];
}

// With generics — one function, type-safe
function getFirst<T>(arr: T[]): T | undefined {
  return arr[0];
}

const firstNum = getFirst([1, 2, 3]);     // TypeScript infers T = number
const firstStr = getFirst(["a", "b"]);    // TypeScript infers T = string
const firstUser = getFirst<User>([user1, user2]);  // explicit

// Real-world example: API response wrapper
interface ApiResponse<T> {
  data: T;
  error: string | null;
  loading: boolean;
}

type UserResponse = ApiResponse<User>;
type UserListResponse = ApiResponse<User[]>;
```

### 3. Utility Types — TypeScript 内置工具

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  password: string;
}

// Partial — all fields optional (useful for PATCH requests)
type UserUpdate = Partial<User>;
// { id?: number; name?: string; email?: string; password?: string }

// Pick — select specific fields
type PublicUser = Pick<User, "id" | "name">;
// { id: number; name: string }

// Omit — exclude specific fields
type SafeUser = Omit<User, "password">;
// { id: number; name: string; email: string }

// Readonly — immutable
type ImmutableUser = Readonly<User>;
// Can't reassign any field

// Record — map type
type UserMap = Record<string, User>;
// { [key: string]: User }
```

## 猜猜输出 / What Does This Output?

```typescript
type Status = "active" | "inactive";

function updateUser<T extends { id: number }>(
  user: T,
  updates: Partial<T>
): T {
  return { ...user, ...updates };
}

const user = { id: 1, name: "Alice", status: "active" as Status };
const updated = updateUser(user, { name: "Bob" });
console.log(updated);
```

**A)** `{ id: 1, name: "Bob", status: "active" }`
**B)** `{ id: 1, name: "Bob" }` — only updated fields
**C)** TypeScript compilation error
**D)** `{ name: "Bob" }` — overwrites everything

<details><summary>显示答案 / Show Answer</summary>

**A** — `{ id: 1, name: "Bob", status: "active" }`

The spread `{ ...user, ...updates }` keeps all original fields and only overwrites `name`. TypeScript infers `T = { id: number, name: string, status: Status }` from the argument, and `Partial<T>` means all fields in `updates` are optional.

`T extends { id: number }` is a **generic constraint** — it says "T must have at least an `id: number` field." This lets you write type-safe functions that work with any object shaped that way.

</details>

## ❌ vs ✅ Common Mistakes

**❌ Using `any` everywhere:**
```typescript
function processData(data: any): any {
  return data.someField;  // No type safety — defeats the purpose!
}
```

**✅ Use generics or specific types:**
```typescript
function processData<T extends { someField: string }>(data: T): string {
  return data.someField;  // Type-safe!
}
```

**❌ Ignoring `null` and `undefined`:**
```typescript
function getUserName(user: User): string {
  return user.name.toUpperCase();  // Crashes if name is null!
}
```

**✅ Handle nullable types:**
```typescript
function getUserName(user: User): string {
  return user.name?.toUpperCase() ?? "Unknown";
}
```

## 何时用 / When to Use

✅ **用 TypeScript 当：**
- 团队 > 2人的任何项目
- 公共 API / 组件库（类型就是文档）
- 需要重构的大型代码库

⚠️ **小心过度工程：**
- 不要为了类型安全而写出复杂到没人看得懂的类型体操
- `unknown` 代替 `any`，但不要到处用高级类型体操

## 🧒 ELI5

TypeScript 的类型就像乐高积木的形状说明。如果积木形状不对，在你拼之前就会报错，而不是等你建好了整栋房子才发现某块积木装不进去。

泛型就像一个可以适配任何形状的万能模具 — 你告诉模具"我要方形的"，它就给你方形的东西，类型安全全程保证。

TypeScript types are like LEGO piece shape guides. If a piece doesn't fit, it tells you *before* you try to assemble, not after you've built the whole house.

Generics are like a universal mold that adapts to any shape you specify — tell it "I want squares," it gives you type-safe squares every time.
