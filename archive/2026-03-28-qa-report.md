## ⚠️ Issue 1: Architecture Diagram Contradiction (Significant)

**Location:** Part 2 — Architecture Diagram + Step 1-3 code

**The problem:**
The diagram shows a SINGLE consumer group where:
- Consumer 0 → Email Worker
- Consumer 1 → SMS Worker
- Consumer 2 → Push Worker

But the partition key is `user_id` (via `key=event.user_id.encode()`). In Kafka, a single consumer group partitioned by `user_id` means:
- Consumer 0 gets ALL events (order.placed, payment.failed, shipment.updated) for users where `user_id % 3 == 0`
- Consumer 1 gets ALL event types for users where `user_id % 3 == 1`
- etc.

So each consumer would actually handle email + SMS + push for its assigned users — NOT just one channel. The diagram implies channel-based routing (Consumer 0 = email only), which contradicts how Kafka consumer groups actually work.

**What the correct architecture looks like:**
Option A (what the code's `_handler_map` routing actually implements): ONE consumer group where each consumer instance routes internally to email/SMS/push based on event_type. The diagram should show consumers → routing logic → email/sms/push workers.

Option B (true pub/sub): THREE separate consumer groups (`email-notification-group`, `sms-notification-group`, `push-notification-group`), each consuming the full topic independently. This is how most real systems do it.

**Severity:** This is a meaningful technical inaccuracy that would confuse a reader trying to implement the design. However, the code's `_handler_map` routing pattern (correct for Option A) and the interview Q&A section (accurate) save the overall educational value.
